# ASCII Art 스킬

UI 레이아웃과 데이터 흐름을 이슈 본문에서 바로 읽을 수 있는 ASCII Art로 만든다. 구현 세부 설계가 아니라 요구사항 합의를 돕는 대략도에 집중한다.

## 작성 원칙

- GitHub Markdown에서 깨지지 않도록 fenced code block 안에 작성한다.
- 선은 ASCII 문자만 사용한다: `+`, `-`, `|`, `/`, `\`, `>`, `<`, `[`, `]`.
- 정렬이 중요한 도식 안에는 긴 한국어 문장을 넣지 않는다.
  - 박스 안은 짧은 영어/숫자/약어를 우선 사용한다.
  - 필요한 설명은 도식 아래 한국어 bullet로 보충한다.
- 실제 구현이 확정되지 않은 영역은 `?`, `(TBD)`, `[optional]`처럼 표시한다.
- UI 대략도는 실제 pixel-perfect 설계가 아니라 영역, 우선순위, 상호작용 위치를 보여준다.
- 데이터 흐름도는 source, transform, storage, output, error path를 구분한다.

## UI 레이아웃

UI 레이아웃 변경 이슈에는 `## 예상 UI 레이아웃` 섹션을 추가하고 다음 순서로 작성한다.

1. 주요 viewport를 정한다: desktop, tablet, mobile 중 변경 영향이 있는 것만 그린다.
2. header/sidebar/content/footer 같은 큰 영역을 먼저 배치한다.
3. 버튼, 입력, 테이블, 목록, 상세 패널, modal 등 작업 대상 요소만 표시한다.
4. 도식 아래에 사용자 동선과 responsive 차이를 짧게 적는다.

예시:

```text
Desktop
+--------------------------------------------------+
| Header                                           |
+----------+---------------------------------------+
| Sidebar  | Filter [____] [Search] [Add]          |
|          +---------------------------------------+
|          | List                | Detail          |
|          | +----------------+  | +------------+  |
|          | | Row            |  | | Form       |  |
|          | | Row selected   |  | | Actions    |  |
|          | +----------------+  | +------------+  |
+----------+---------------------------------------+

Mobile
+------------------------------+
| Header                       |
+------------------------------+
| Filter [____] [Search]       |
+------------------------------+
| List                         |
| +--------------------------+ |
| | Row                      | |
| +--------------------------+ |
+------------------------------+
| Detail opens as full screen  |
+------------------------------+
```

- Desktop은 목록과 상세를 같은 화면에 배치한다.
- Mobile은 상세를 full-screen 전환 또는 bottom sheet로 연다.

## 데이터 흐름도

데이터 흐름이나 처리 경계가 이슈 이해에 중요하면 `## 데이터 흐름도` 섹션을 추가한다.

```text
[Client]
   |
   | request DTO
   v
[API Controller] ---> [Validation Error]
   |
   | domain command
   v
[Service] ---> [Repository] ---> [DB Table]
   |
   | response DTO
   v
[Client UI]
```

- 화살표 옆에는 데이터 형태나 주요 식별자를 짧게 적는다.
- 저장소/외부 시스템/API 경계는 별도 박스로 분리한다.
- 실패 흐름이 요구사항에 중요하면 success path와 error path를 함께 표시한다.

## 이슈 본문 삽입 규칙

- UI 레이아웃이 바뀌면 `작업 계획` 앞이나 뒤에 `## 예상 UI 레이아웃`을 둔다.
- 데이터 계약, replay, socket, batch, migration처럼 흐름이 중요한 작업은 `## 데이터 흐름도`를 둔다.
- 도식이 너무 커지면 핵심 happy path만 본문에 넣고 세부 흐름은 하위 이슈나 문서 작업으로 분리한다.
